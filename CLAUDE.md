## 프로젝트 개요

**drum-room (드럼룸)** — 드럼 연습 스튜디오 웹앱.

원곡 음원 파일을 넣으면 브라우저 안에서 AI가 드럼 트랙을 분리하고,
사용자는 원곡을 재생하면서 그 안의 드럼 볼륨을 0~100%로 조절하며
그 위에 직접 드럼을 친다.

**핵심 — 오해 금지:**
- "드럼을 제거한 반주(MR)"를 만드는 도구가 아니다.
- 원곡을 재생하면서 그 안의 드럼 볼륨만 조절하는 것이 목적이다.
- 드럼 0%면 드럼 없이, 25%면 가이드로 작게 깔고, 100%면 원곡 그대로.

---

## 1차 버전 범위 (지금 만드는 것)

기능은 딱 4개. 이 외에는 1차에 없다.

1. 곡 파일 넣기 (드래그 입력)
2. 드럼 분리 (브라우저 내, 곡당 1회, 몇 분 소요)
3. 드럼 볼륨 슬라이더 0~100% (+ 프리셋: 드럼없이/가이드/원곡)
4. 재생 / 정지

### 2차 이후로 미룬 것 (1차에 넣지 않음)
- 구간 루프, 템포 조절, 메트로놈 — 실시간 정밀 오디오 처리. 가장 어려운 영역.
- 악보 PDF 표시
- 분리 결과 영구 저장 (1차는 세션 동안만 메모리 유지)
- 모바일·Safari 등 타 브라우저 지원

미룬 것은 폐기가 아니다. 1차를 써본 뒤 순서대로 붙인다.

## 기술 스택

- **프레임워크:** Next.js 16 (App Router, 정적 export)
- **호스팅:** Vercel 정적 호스팅 (서버 비용 0원, 곡 파일이 서버로 안 감)
- **스타일:** Tailwind
- **분리 엔진:** Demucs v4(htdemucs) → ONNX, onnxruntime-web 으로 브라우저 실행
  - 모델 가중치 약 81MB, 첫 방문 1회 다운로드 후 캐시
  - 분리는 곡을 세그먼트 단위로 잘라 순서대로 처리 (메모리 안전 — OOM 방지)
  - 분리 추론은 Web Worker에서 수행 (메인 스레드/UI 안 막힘)
- **오디오 재생:** Web Audio API. 드럼 트랙 / 드럼외 트랙 두 개를 동시 재생,
  드럼 트랙에만 게인 노드(볼륨 슬라이더) 연결.
- **폰트:** Pretendard

### 1차 지원 타깃
- 크롬 데스크톱 전용. 모바일·Safari 등은 접속 시 안내 화면만 표시.

## 작업 규칙 (standing instructions)

- **Phase A(설계 점검) → 승인 → Phase B(구현)** 순서를 지킨다.
  설계 변경이 필요하면 먼저 점검 결과를 제시하고 승인을 받는다.
- **다운그레이드·우회·타협 금지.** 막히면 더 나은 솔루션을 근거와 함께 제시한다.
  품질 우선, 속도·범위는 양보 가능 — 단, 품질은 양보하지 않는다.
- **작은 이상도 그냥 넘기지 않는다.** 이상 징후는 root cause까지 추적한다.
- 객관식으로 떠넘기지 않는다. 근거를 갖춘 단일 최선안을 제시한다.
- 주요 아키텍처·설계 결정은 외부 모델 교차검증을 거친 뒤 확정한다.
- 진행률은 "현 시점 best baseline" / "1차 cycle 마무리" framing 사용.
  "완료/종결/확정" 같은 단정 표현은 쓰지 않는다.

## 검증 기록 (1차 설계 확정 전 외부 검증 완료)

1차 설계는 Gemini·ChatGPT 교차검증을 거쳤다. 핵심 결론:
- **재생 엔진이 이 제품의 진짜 난이도다.** 분리(ML)는 느려도 되는 배치 작업.
  실시간 재생·동기화가 DAW 수준 정밀도를 요구함.
- 그래서 1차는 실시간 정밀 처리(루프·템포·메트로놈)를 빼고,
  "곡 두 개 동시 재생 + 한쪽 볼륨 조절" 만 한다 — 브라우저가 원래 잘하는 일.
- 2차에서 실시간 영역에 들어갈 때를 위한 검증 결과:
  - 음정 보존 템포 조절은 Rubber Band WASM 사용 (SoundTouchJS 아님).
  - 메트로놈은 3-clock 구조(실제시간/늘어난시간/박자시간)로 설계.
  - 재생 중 메모리 할당 제로, GC 글리치 방지.
  - 샘플레이트 불일치·블루투스 지연 주의.
